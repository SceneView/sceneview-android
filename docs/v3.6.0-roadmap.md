# SceneView v3.6.0 — API Simplification Roadmap

> Goal: make every API so clear that an AI can generate correct, complete code on the first try.

## Audit findings (14 issues)

Based on a full audit of the public API surface across Android (`sceneview` + `arsceneview`),
Apple (`SceneViewSwift`), and KMP core (`sceneview-core`).

---

## Priority 1 — Breaking / Confusing API patterns

### 1. `LightNode` dual `apply` lambdas → merge into one
**Problem:** `LightNode(type, apply: Builder.() -> Unit, nodeApply: LightNodeImpl.() -> Unit)` —
two separate `apply` lambdas with different receivers. No other node has this. An AI always picks
the wrong one.

**Fix:** Merge into a single `apply: LightNode.() -> Unit` that exposes both builder config
(via delegated properties like `intensity`, `color`, `direction`) and node config (position, etc.).
The `LightManager.Builder` should be an implementation detail.

```kotlin
// Before (confusing):
LightNode(
    type = LightManager.Type.DIRECTIONAL,
    apply = { intensity(100_000f); direction(0f, -1f, 0f) },  // Builder receiver
    nodeApply = { position = Position(0f, 5f, 0f) }           // Node receiver
)

// After (v3.6.0):
LightNode(
    type = LightManager.Type.DIRECTIONAL,
    intensity = 100_000f,
    direction = Direction(0f, -1f, 0f),
    position = Position(0f, 5f, 0f)
)
```

**Migration:** Deprecate `apply`/`nodeApply` params, add explicit params for common properties.

---

### 2. `CameraNode` composable ambiguity → rename to `SecondaryCamera`
**Problem:** `CameraNode { }` inside `SceneScope` creates a non-active camera. The *active* camera
is the `cameraNode` parameter of `Scene(...)`. Same name, different behavior. An AI generates
`CameraNode { }` thinking it activates.

**Fix:** Rename the composable to `SecondaryCamera` or `CameraViewpoint`. Add clear docs.

```kotlin
// Before (ambiguous):
Scene(cameraNode = rememberCameraNode(engine)) {
    CameraNode { /* non-active camera — confusing */ }
}

// After (v3.6.0):
Scene(cameraNode = rememberCameraNode(engine)) {
    SecondaryCamera { /* explicitly named as non-primary */ }
}
```

---

### 3. Geometry composables — inconsistent transform params
**Problem:** `CubeNode` has `position`+`rotation` but no `scale`. `SphereNode` has `position` but
no `rotation`. `CylinderNode`/`PlaneNode` have both. Inconsistent — AI has to guess which params
exist.

**Fix:** ALL geometry composables get the same transform trio: `position`, `rotation`, `scale`.

```kotlin
// v3.6.0 — uniform signature for all geometry nodes:
CubeNode(size, materialInstance, position, rotation, scale, apply, content)
SphereNode(radius, materialInstance, position, rotation, scale, apply, content)
CylinderNode(radius, height, materialInstance, position, rotation, scale, apply, content)
PlaneNode(size, materialInstance, position, rotation, scale, apply, content)
```

---

### 4. `ModelNode` — `scaleToUnits` silently overrides `scale`
**Problem:** Passing both `scaleToUnits = 0.5f` and `scale = Scale(2f)` → `scale` is silently
ignored. No warning, no error.

**Fix:** Make them mutually exclusive at the type level OR document with `require()` check.

Option A (preferred): Change `scaleToUnits` to a modifier function:
```kotlin
ModelNode(modelInstance = instance)
    .scaleToFit(units = 0.5f)  // explicit, separate step
```

Option B: Runtime warning when both are set.

---

## Priority 2 — Missing composables & convenience APIs

### 5. Add `PhysicsNode` and `ShapeNode` composables to `SceneScope`
**Problem:** `PhysicsNode.kt` and `ShapeNode.kt` exist as classes but have no corresponding
`@Composable` in `SceneScope`. AI can't discover them through the DSL.

**Fix:** Add composable wrappers following the same pattern as `CubeNode`, etc.

---

### 6. `VideoNode` + `MediaPlayer` friction → add convenience
**Problem:** `VideoNode` requires a `MediaPlayer` that must be separately created, prepared, and
started. `rememberMediaPlayer` exists but isn't well-documented.

**Fix:** Add `rememberVideoNode(assetFileLocation)` that bundles player lifecycle:
```kotlin
Scene {
    VideoNode(videoPath = "videos/intro.mp4", autoPlay = true)
}
```

---

### 7. `ReflectionProbeNode` — bring into `NodeLifecycle` pattern
**Problem:** Uses a standalone composable function, no `apply` or `content` params, doesn't
participate in `NodeLifecycle`. Inconsistent with every other node.

**Fix:** Refactor to match the standard pattern:
```kotlin
ReflectionProbeNode(
    environment = env,
    position = Position(0f, 1f, 0f),
    radius = 5f,
    apply = { priority = 1 },
    content = { /* child nodes affected by this probe */ }
)
```

---

## Priority 3 — Cross-platform parity

### 8. Swift `SceneView` — move from imperative to declarative
**Problem:** Android's `Scene { ModelNode(...) }` is declarative and reactive. Swift's
`SceneView { root in root.addChild(model.entity) }` is imperative (called once). An AI generating
cross-platform code produces structurally different code.

**Fix:** Implement a `@resultBuilder`-based DSL or SwiftUI `ViewBuilder` pattern:
```swift
// Target API (v3.6.0):
SceneView {
    ModelNode("models/helmet.usdz")
        .scaleToFit(0.5)
        .position(x: 0, y: 0, z: -2)

    GeometryNode.cube(size: 0.3, color: .red)
        .position(x: 1, y: 0, z: -2)
}
```

`NodeBuilder` already exists — integrate it as the default content builder for `SceneView`.

---

### 9. Swift `NodeGesture` — add automatic cleanup
**Problem:** `NodeGesture` uses global static storage by `ObjectIdentifier`. Handlers are never
automatically cleaned up. AI won't know to call cleanup.

**Fix:** Use `Entity.components` or weak references for automatic cleanup. Alternatively, add
`.onDisappear` lifecycle integration.

---

### 10. Align Swift node types with Android composable names
**Problem:** Android has `CubeNode`, `SphereNode`, `CylinderNode`, `PlaneNode` as separate
composables. Swift has `GeometryNode.cube()`, `GeometryNode.sphere()` as static factories.

**Decision:** Keep both patterns (they're idiomatic for each platform) but ensure the same
capabilities exist. Document the mapping in `llms.txt`.

---

## Priority 4 — Performance & correctness

### 11. Geometry composables — add equality guards on `SideEffect`
**Problem:** `CubeNode`, `SphereNode`, etc. call `node.updateGeometry(...)` on every recomposition
via `SideEffect`, even when geometry params haven't changed.

**Fix:** Add `remember(size, cornerRadius) { }` guards or use `derivedStateOf`.

---

### 12. `HitResultNode` — add simple convenience overload
**Problem:** 11 configurable parameters. Most users need `(xPx, yPx)` → `HitResult?`.

**Fix:** Add a `SimplifiedHitResult` composable:
```kotlin
HitResultNode(screenX = 540f, screenY = 960f) { hitResult ->
    ModelNode(modelInstance = instance, position = hitResult.position)
}
```

---

## Priority 5 — Architecture alignment

### 13. `sceneview-core` `SceneNode` → integrate with Android `Node`
**Problem:** KMP `SceneNode` interface is defined but Android `Node` doesn't implement it.
The abstraction is aspirational, not functional.

**Fix:** Make Android `Node` implement `SceneNode`. Requires aligning property names and types.
This enables true shared logic in `sceneview-core`.

---

### 14. `ARSceneScope` nodes only available at root level
**Problem:** AR composables (`AnchorNode`, etc.) are on `ARSceneScope`, not `NodeScope`. Nesting
AR nodes inside `Node { }` doesn't compile. Error message is confusing.

**Fix:** Add `ARNodeScope` that extends `NodeScope` with AR composables. Or document this
limitation prominently in `llms.txt` with the correct pattern.

## Migration strategy

- All breaking changes use `@Deprecated(replaceWith = ...)` for one minor version
- Kotlin compiler will auto-suggest fixes via IDE quick actions
- `llms.txt` updated per alpha release so AI tools always generate current code
- Migration guide in `docs/docs/migration.md`

## Success criteria

An AI assistant should be able to:
1. Generate correct `Scene { }` code with any node type without consulting `apply` blocks
2. Generate correct `ARScene { }` code with nested AR nodes
3. Generate cross-platform code (Android + Swift) with structurally similar patterns
4. Never hit a "silent override" or "wrong receiver" trap
