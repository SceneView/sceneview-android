The infrastructure

Five layers — each one permanent, each one compounding

CLAUDE.md Project instructions checked into the repo. Architecture decisions, threading rules, API conventions, known gotchas. Loaded automatically into every session — no re-explaining across conversations or team members.
llms.txt Machine-readable API reference at the repo root. Any AI reads it and instantly becomes a competent user of the SDK. Zero integration friction for any AI-assisted contributor or customer.
MCP server Exposes docs, live GitHub issues, code validator, and working samples as callable tools. Published npm package — integrates in any MCP-compatible AI IDE or assistant.
/skills (slash cmds) Custom Claude Code workflows: /review, /document, /test, /contribute. The full contribution pipeline runs autonomously. Standardised across all contributors — human or AI.
Auto-memory Persistent memory files per project: decisions made, bugs found and fixed, domain-specific traps. The AI accumulates knowledge across sessions — no context reset means no repeated mistakes.

Honest assessment — the limits

Where human judgment remained essential

Build system hallucinations
Confidently wrote dependency versions that don't exist: Kotlin 2.3.10, AGP 9.1.0, Filament 1.70.0. Each broke the build silently. Fix: a strict version catalog as the single source of truth, verified against real published releases.
Native/binary API surface gaps
Applied Filament 1.57.x patterns to a 1.56.x codebase — causing silent SIGABRT crashes on dispose. AI doesn't know what it doesn't know about compiled native libraries. Domain expertise to catch this is non-negotiable.
Hardware-specific bugs
The new Compose Scene rendered pure black on the Apple M3 OpenGL ES emulator. Required several sessions to diagnose — a surface lifecycle issue specific to that config. Without a physical device to reproduce, AI reasoning alone isn't sufficient.
Context loss between sessions
Each conversation started cold. Without intentionally-built memory files and CLAUDE.md, the same mistakes recurred. The infrastructure (slide 5) solves this — but it requires discipline to set up and maintain. It's not automatic.

Applying this to our repos

A prioritised action plan for Octopus Community

1
Add CLAUDE.md to each active repo. Architecture decisions, API conventions, threading constraints, known gotchas. This is the foundation. Everything else builds on it. Time: 1–2h per repo.
2
Write an llms.txt for each public SDK surface. Machine-readable API reference. Any AI — ours and any customer's — becomes a competent user instantly. Direct competitive differentiation for an SDK company.
3
Define shared /skills (slash commands). /review, /document, /test, /contribute. Standardise the contribution pipeline across all repos. AI can run it autonomously and consistently.
4
Ship an MCP server for the Octopus SDK. Expose our docs, integration samples, changelog, and live issues as MCP resources. Our SDK becomes a first-class citizen in AI-powered IDEs — for our whole developer audience.
5
Strict version catalog per repo. The #1 source of AI build errors is hallucinated dependency versions. One catalog file, verified against real releases. Pin everything.
6
Maintain memory files for domain-specific gotchas. Native API quirks, platform-specific bugs, lifecycle traps. AI encounters them repeatedly without this. Write them down once; eliminate recurring mistakes permanently.
